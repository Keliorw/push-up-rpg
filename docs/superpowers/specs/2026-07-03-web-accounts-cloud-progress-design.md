# Design: аккаунты (логин + пароль) и облачный прогресс для веб-версии

**Дата:** 2026-07-03
**Область:** `web-game/` (статический сайт на GitHub Pages)
**Статус:** утверждён к реализации

## 1. Проблема

Сейчас прогресс веб-версии (`web-game`) хранится в `localStorage`
(`web-game/src/storage.ts`, ключ `pushuprpg.progression`). Он привязан к
конкретному браузеру: теряется при смене устройства, очистке данных, заходе
из другого браузера. Нужны учётные записи, чтобы прогресс привязывался к
пользователю и синхронизировался между устройствами. В перспективе — таблица
рейтинга на экране «Арена».

## 2. Цели

- Учётная запись по **логину (никнейм) и паролю**.
- Прогресс хранится в облаке и синхронизируется между устройствами.
- Сессия сохраняется надолго (до явного выхода).
- Модель данных, готовая к будущей таблице рейтинга (ник + счёт).

## 3. Вне области (non-goals)

- Вход через Google / соц-сети.
- Восстановление пароля и e-mail (подтверждено: e-mail не нужен).
- Серверный анти-чит / валидация счёта (Cloud Functions).
- Сам UI таблицы рейтинга на Арене — только фундамент под неё.
- Гостевой режим — **аккаунт обязателен** (подтверждено).

## 4. Технологический выбор

**Firebase — Authentication (Email/Password) + Cloud Firestore.**

Обоснование (прототип «для себя и друзей», приоритет — простота и бесплатность):

- Бесплатный план Firebase **не засыпает** при простое (в отличие от Supabase,
  где проект встаёт на паузу после 7 дней без активности).
- Весь обмен — из клиентского JS, свой сервер не нужен (GitHub Pages — статика).
- На объём данных (несколько полей на пользователя) free-tier с огромным запасом.
- Ключи Firebase-конфига публичны by design; безопасность обеспечивается
  Security Rules, а не секретностью ключей.

## 5. Аутентификация

Провайдер — **Email/Password**. Пользователь вводит **никнейм** и пароль;
про e-mail не знает.

- **Маппинг ника на технический e-mail:** `<normalized-nick>@pushuprpg.app`.
  Нормализация ника: нижний регистр, разрешённые символы `a-z0-9_-`, длина
  3–20. Ввод, не проходящий валидацию, отклоняется с понятным сообщением до
  обращения к Firebase.
- **Регистрация** (первый вход): ник + пароль (мин. 6 символов — требование
  Firebase). Уникальность ника обеспечивает Firebase: повтор технического
  e-mail → ошибка `auth/email-already-in-use` → показываем «Логин уже занят».
- **Вход** (возврат / новое устройство): ник + пароль.
- **Отображаемое имя:** оригинальный (не нормализованный) ник сохраняем как
  `nickname` в документе пользователя и в `displayName` — он пойдёт в рейтинг.
- **Персистентность сессии:** `browserLocalPersistence` (по умолчанию в web SDK).
  Пользователь остаётся залогиненным **до явного «Выйти»**; ID-токен
  обновляется автоматически по refresh-токену. Отдельный «срок на год»
  настраивать не требуется — фактически дольше.

### Известное ограничение

Без e-mail **нет восстановления пароля**: забытый пароль = потерянный аккаунт и
прогресс. Принято осознанно (прототип). Добавление опционального e-mail
возможно позже без ломки модели.

## 6. Модель данных (Firestore)

Коллекция `users`, документ с ключом `uid` пользователя:

```
users/{uid} = {
  nickname:        string,   // оригинальный ник (для показа и рейтинга)
  defeatedCount:   number,   // 0..MONSTER_SEQUENCE.length, целое
  lastWorkoutDate: string|null, // ISO-дата последней тренировки
  updatedAt:       number    // серверное время последней записи
}
```

Это надмножество текущего `Progression` (`defeatedCount`, `lastWorkoutDate`) +
`nickname` и `updatedAt`. Модель сразу «рейтинг-готова».

### Security Rules (черновик)

```
match /users/{uid} {
  // Публичное чтение — для будущего рейтинга. Секретов в документе нет.
  allow read: if true;

  // Писать может только владелец; ник неизменяем после создания;
  // defeatedCount — целое >= 0.
  allow create: if request.auth != null && request.auth.uid == uid
                && request.resource.data.defeatedCount is int
                && request.resource.data.defeatedCount >= 0;
  allow update: if request.auth != null && request.auth.uid == uid
                && request.resource.data.nickname == resource.data.nickname
                && request.resource.data.defeatedCount is int
                && request.resource.data.defeatedCount >= 0;
}
```

(Точные правила уточняются при реализации; принцип: чтение публичное, запись —
только владелец, ник неизменяем.)

## 7. Модули (`web-game/src/`)

Разбиение с чёткими границами; каждый модуль тестируем/понятен отдельно.

- **`firebase.ts`** — инициализация Firebase App, Auth, Firestore. Конфиг
  подключается как ESM с CDN gstatic. Единственная точка, знающая о конфиге.
- **`auth.ts`** — тонкая обёртка над Firebase Auth:
  `register(nick, password)`, `login(nick, password)`, `logout()`,
  `onUser(cb)` (подписка на состояние входа). Внутри — маппинг ника в e-mail и
  нормализация/валидация. Наружу отдаёт `{ uid, nickname }` или ошибку.
- **`remote-storage.ts`** — работа с Firestore:
  `loadRemote(uid): Promise<Progression|null>`,
  `saveRemote(uid, progression, nickname): Promise<void>` над `users/{uid}`.
- **`sync.ts`** — **чистая** функция `mergeProgress(local, remote)`. Вся логика
  разрешения конфликтов здесь (см. §9). Покрывается юнит-тестом.
- **`storage.ts`** (существующий) — остаётся локальным кэшем (`localStorage`).
- **UI (экран логина/регистрации + чип аккаунта)** — новый экран `screen-auth`
  с полями ник/пароль и переключением регистрация/вход; в меню — ник + «Выйти».
- **`main.ts`** (существующий) — оркестрация: связывает auth, remote, sync и
  экраны (см. §8).

## 8. Потоки

- **Первый вход / нет сессии:** показываем `screen-auth` до входа (аккаунт
  обязателен, гостевого режима нет). После регистрации: создаём `users/{uid}`.
  **Миграция:** если в этом браузере остался локальный прогресс от версии
  *до* аккаунтов (текущий `localStorage`), вливаем его в новый аккаунт через
  `mergeProgress`, чтобы ранние игроки не потеряли прогресс. Затем в меню.
- **Возврат (сессия восстановлена):** сразу меню; при старте тянем remote,
  мёржим с локальным кэшем, результат делаем активным и пишем в оба хранилища.
- **Победа над монстром:** `progression` обновляется → пишем в `localStorage`
  (кэш) и в Firestore.
- **Новое устройство:** вход по нику/паролю → `loadRemote` → прогресс подтянулся.
- **Выход:** очищаем сессию и UI-состояние, возвращаемся на `screen-auth`.

## 9. Разрешение конфликтов (`mergeProgress`)

Чистая функция, «прогресс никогда не откатывается»:

- `defeatedCount = max(local.defeatedCount, remote.defeatedCount)`
- `lastWorkoutDate = максимальная (самая свежая) из двух непустых дат`

Так гарантируется, что при рассинхроне двух устройств берётся наибольший
прогресс, а правило «1 бой в день» опирается на самую свежую дату тренировки.

## 10. Обработка ошибок / оффлайн

- Сбой сети или Firestore при чтении/записи → работаем на `localStorage`,
  показываем ненавязчивую плашку «не синхронизировано, сохранено локально»;
  повтор синхронизации при следующем `save`/`load`.
- Ошибки входа — понятные тексты: «логин занят», «неверный логин или пароль»,
  «слишком короткий пароль», сетевые ошибки.

## 11. Сборка и деплой

- Firebase подключаем как **модульный ESM с CDN gstatic** (как сейчас TF.js в
  `index.html`). В esbuild-сборке пометить CDN-импорты внешними
  (`--external:https://*`), чтобы билд остался **без `npm install`** и выдавал
  тот же `web-game/app.js`.
- Firebase-конфиг (`apiKey` и т.п.) кладём в `firebase.ts` — эти значения
  публичны by design.
- Pages-workflow (`.github/workflows/deploy-pages.yml`) не меняется: он по-
  прежнему грузит папку `web-game/` как артефакт.

## 12. Настройка Firebase-проекта (делает владелец — нужен его аккаунт)

Пошагово (детализируется в плане реализации):

1. Создать проект в Firebase Console.
2. Включить провайдер **Email/Password** в Authentication.
3. Создать базу **Cloud Firestore** (production mode).
4. Выставить Security Rules из §6.
5. Добавить Web App, скопировать объект конфигурации → передать для
   `firebase.ts`.
6. В Authentication → Settings добавить домен `keliorw.github.io` в список
   authorized domains (на случай, если понадобится).

## 13. Готовность к рейтингу (будущий шаг, не сейчас)

Модель данных уже поддерживает запрос
`users` `orderBy(defeatedCount desc) limit N` для таблицы на Арене. Реализация
UI рейтинга — отдельная задача.

⚠️ Так как счёт пишется из браузера, он технически поддаётся накрутке. Для игры
с друзьями это приемлемо; честный анти-чит потребует серверной валидации
(вне текущего скоупа).

## 14. Тестирование

- **Юнит:** `mergeProgress` (чистая логика; ключевой инвариант — прогресс не
  откатывается; корректный выбор свежей даты).
- **Ручное:** регистрация → выход → вход; синхрон между двумя браузерами;
  неверный пароль; занятый логин; оффлайн-фолбэк на localStorage.

## 15. Принятые решения (ранее открытые вопросы)

- Логин — **никнейм + пароль** (не e-mail, не Google).
- **E-mail и восстановление пароля не нужны.**
- **Аккаунт обязателен** с первого входа; гостевого режима нет.
- Масштаб — прототип для себя и друзей; бесплатный tier.
