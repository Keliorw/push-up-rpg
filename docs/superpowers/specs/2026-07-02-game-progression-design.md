# Дизайн: игровая прогрессия (карта, враги, циклы тренировок)

Дата: 2026-07-02
Статус: на ревью
Ветка: push-ups-tracking-mvp

## Контекст

push-up-rpg превращает реальные отжимания в игровой прогресс. Детекция отжиманий
(MoveNet + RepDetector) уже работает и проверяется через браузерный `web-demo`.
Этот документ описывает **игровой луп**: путь по карте из 10 локаций, где каждый
монстр побеждается тренировкой (отжиманиями), а отжимания = XP монстра.

Ассеты (в `~/Downloads/games`): одна картинка карты (768×1376) с 10 нумерованными
узлами-локациями и 12 готовых карточек врагов (локации 1–3: босс + 3 приспешника,
арт и текст впечатаны в картинку 1024×1024). Арта для боссов 4–10 пока нет.

## Скоуп

**MVP — локации 1–3 (12 монстров, циклы 1–3).** Модель данных и прогрессия
рассчитаны на 10 локаций, но заполнены только 1–3; узлы 4–10 на карте
заблокированы. Расширение до 10 = добавление данных + карточек, без изменения
логики.

Решения, зафиксированные с пользователем:
- Строим **общее ядро (чистый TS) + играбельный web-прототип**. RN-приложение
  переиспользует ядро позже (в этот скоуп RN-экраны НЕ входят).
- Контент MVP — только локации 1–3.

## Архитектура

Два слоя, соединённые через явные интерфейсы:

- **`app/src/game/` — игровое ядро (чистый TS, без UI, покрыто Jest).** Данные
  монстров/локаций, прогрессия, правило «1 бой в день», логика подхода/сетов, XP.
  RN-приложение сможет переиспользовать его без изменений.
- **`web-game/` — браузерный прототип.** Экраны (карта, карточка, тренировка,
  победа) на HTML/canvas. Переиспользует `app/src/game/*` И существующий детектор
  `app/src/pose/*` (MoveNet + RepDetector, как в `web-demo`). Собирается esbuild.
  `web-demo` (чистая демка детекции) остаётся нетронутым.

## Модель данных (`app/src/game/types.ts`)

```
type MonsterKind = 'minion' | 'boss';
interface Monster {
  id: string;            // 'loc1-m1', 'loc1-boss', ...
  name: string;
  kind: MonsterKind;
  cardImage: string;     // путь относительно games/, напр. '1/kaban.png'
  sets: number;          // приспешник: 1; босс: 3
  repsPerSet: number;    // норма отжиманий в подходе
  restBetweenSetsSec: number; // отдых между сетами (0 для локаций 1–3)
}
interface Location {
  index: number;         // 1..10
  name: string;
  nodePos: {x: number; y: number}; // доля от размера карты (0..1) — центр узла
  monsters: Monster[];   // [m1, m2, m3, boss]
  locked: boolean;       // true для 4..10 в MVP
}
```

- `MONSTER_SEQUENCE: Monster[]` — плоская последовательность из 12 монстров
  (порядок прохождения), выведенная из `LOCATIONS`.
- Итоговая «HP» монстра = `sets * repsPerSet` (XP = число отжиманий).

## Игровая логика (юниты)

### progression (`app/src/game/progression.ts`)
- `Progression { defeatedCount: number; lastWorkoutDate: string | null }`
  (`lastWorkoutDate` — ISO-дата `YYYY-MM-DD`).
- `currentMonster(p): Monster | null` — `MONSTER_SEQUENCE[defeatedCount]`
  (null, если всё пройдено).
- `defeatMonster(p, today): Progression` — `defeatedCount+1`,
  `lastWorkoutDate = today`. Чистая функция.

### dailyLock (`app/src/game/dailyLock.ts`)
- `isLockedToday(p, today): boolean` — `p.lastWorkoutDate === today`
  (после победы над монстром день «занят» до следующей локальной даты).
- Дата инъектируется параметром (для тестов и для dev-панели), не берётся из
  `Date` внутри ядра.

### workout (`app/src/game/workout.ts`)
Конечный автомат одной тренировки над монстром (переиспользует события повторов
от `RepDetector` через слой прототипа — сюда приходят «засчитан повтор»).
- `WorkoutState { setIndex, repsInSet, done }`.
- `newWorkout(monster): WorkoutState`.
- `onRep(state, monster): { state, event }` — инкремент повтора; когда
  `repsInSet === repsPerSet`:
  - последний сет → `event = 'monsterDefeated'`, `done = true`;
  - иначе → `event = 'setComplete'` (прототип показывает экран отдыха
    `restBetweenSetsSec`, затем `setIndex+1`, `repsInSet=0`).
- Прогресс/HP: доля = `(суммарные повторы) / (sets*repsPerSet)`.
- Босс засчитывается только при прохождении всех сетов **за одну сессию**;
  выход посреди боя — при следующем входе бой с начала (состояние не
  персистится между сессиями).

## Экраны и поток (web-прототип)

```
START → Карта → (тап по текущему узлу) → Карточка врага
     → «Начать тренировку» → Тренировка → Победа → Карта (узел засчитан, день залочен)
```

- **Карта** — картинка карты, поверх — маркеры узлов 1–10 в `nodePos`:
  пройденные (✓), текущий (пульсирующий), заблокированные (🔒). Активен только
  текущий узел. Внутри локации 4 боя идут по порядку; узел показывает
  под-прогресс («приспешник 2/3» / «БОСС»).
- **Карточка врага** — полноэкранная карточка (готовый арт), норма
  («Победи: 1×8» или «Босс: 3×8»), HP-бар, кнопка «Начать тренировку». Если день
  залочен — вместо кнопки «Приходи завтра».
- **Тренировка** — камера + скелет + большой счётчик (как в `web-demo`), HP-бар
  врага убывает по повторам; для босса — индикатор «Сет 1/3» и экран отдыха между
  сетами. По достижении нормы → Победа.
- **Победа** — короткий экран «Повержен!», затем возврат на карту с продвижением.
- **Dev-панель** (свернутая, для тестов, только в прототипе): сбросить дневной
  лок, сбросить прогресс, прыгнуть к монстру N. Без неё игру с правилом «1/день»
  невозможно тестировать.

## Хранение и ассеты

- Прогресс — `localStorage` (ключ `pushuprpg.progression`). В RN позже —
  AsyncStorage; логика в ядре одинаковая, хранилище инъектируется.
- Ассеты копируются в репозиторий в `web-game/games/` (карточки ужимаются до
  ~600px, чтобы не раздувать git; оригиналы 1024² тяжёлые). Карта — как есть.

## Тестирование

- `app/src/game/*` — Jest: последовательность и нормы (12 монстров, циклы 1–3),
  `currentMonster`/`defeatMonster`, `isLockedToday` (с инъекцией даты), автомат
  `workout` (миньон 1 сет; босс 3 сета: setComplete×2 → monsterDefeated; XP/доля).
- UI прототипа — визуально в браузере (камера/canvas в Jest не работают).

## Вне скоупа

- RN-экраны игры (ядро переиспользуется позже).
- Арт и контент боссов 4–10.
- Строгий 3-минутный таймер отдыха (это конфиг босса №8, вне MVP; поле
  `restBetweenSetsSec` уже заложено).
- Онлайн/соц-функции, монетизация.

## Допущения

- Привязка приспешник→норма — по порядку из ТЗ (1-й миньон = меньшая норма).
- Координаты узлов (`nodePos`) калибруются по картинке карты вручную.
- Отдых между сетами боссов 1–3 — 0 (пропускаемый); строгие тайминги — позже.
